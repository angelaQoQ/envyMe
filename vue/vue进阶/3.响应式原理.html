# 3.1 发布订阅者模式

如同微信公众号推送每日文章,某公众号管理者先将文章推送到消息通道,然后微信消息通道再把该文章推送给订阅的用户.其中:

  1. 参与角色:
    • 发布者:就是公众号管理者
    • 订阅者:就是订阅的用户
    • 消息通道:
      他们之间并不是直接发送获取数据,而是通过消息通道进行统一管理.

  2. 发布订阅者的
    • 优点:
      • 解耦. 发布者和订阅者没有直接关联,不会直接相互影响,可以独立的管理和维护;
      • 性能. 发布者不需要等待订阅者的成功接受响应,减少很多业务等待时间;
      • 方便管理维护. 发布者和订阅者通过 消息发布结构关联,可以通过 消息通道进行 测试.
    • 缺点:
      • 单向通信造成 发布者无法确定订阅者是否接受到了订阅内容.

  3. 适宜场景:
    1. 数量庞大的订阅者
    2. 发布者需要与多方交互,如vue中 修改数据时 同步内存,同步DOM,同步虚拟DOM
    3. 不关心订阅者的响应

  4. 与观察者模式的区别:
  • 发布订阅是观察者的一种变体
  • 发布订阅使用异步消息队列实现, 观察者使用同步方式实现
  • 发布订阅可以把 发布服务器 , 订阅服务器 部署在不同的应用中

# 3.2 依赖收集

依赖什么意思 ?

  • vm.data中的数据是响应式的
    举栗: vm.data中有个变量叫ABC, 它是响应式的. 但凡是和ABC有关系的,有一个算一个都叫ABC的订阅者

  • ABC怎么个响应式呢?
    简单点说: 用户通过触发事件修改了ABC,页面上所有使用了ABC的Dom都会自动更新.

  • 我们一步步来分析:

    1. 修改vm.data中变量 ---> 会触发vm.data对象中的setter ----> 所以可以在ABC的setter函数中触发DOM更新
    2. 由1引出另外一个问题: 我怎么知道有哪些用过ABC的DOM结构?
      只要用过变量ABC,就会触发ABC的getter,getter里收集对ABC产生依赖的订阅者!,天才!

        get(){
            // 在执行o2.b时,这里会调用
        },
        set(newValue){      //接受唯一参数，即该属性新的参数值。
            // 在执行o2.b = * 时,这里会调用
        }         
    })

# 3.4 响应式原理

一句话: 组件初始化时,Vue遍历对象参数中data属性, 将所有属性/变量添加getter和setter,

  • 在虚拟DOM获取变量值时 利用变量的getter 整理出该变量的所有订阅者,存入数组.
  • 在修改变量时, 触发setter, 遍历该变量的所有订阅者, 并将最新的变量值传给各个订阅者.

getter 整理收集变量的依赖(订阅者)
setter 触发变量依赖的数据更新
